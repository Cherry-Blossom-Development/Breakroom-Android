## USB Switch Issue — Mouse/Keyboard Not Working After Switch

### Problem
After switching to another computer using the 4-way USB switch and switching back, the wireless mouse and keyboard stop working on the Dell Inspiron 16 Plus 7620. The only reliable fix initially was a full reboot.

### Setup
- Laptop: Dell Inspiron 16 Plus 7620, BIOS 1.36.0 (as of 2026-06-17)
- Switch: 4-way USB switch (Genesys Logic hub, VID_05E3&PID_0610)
  - Port 1: This Dell laptop
  - Port 2: Mac Mini
  - Port 3: Another PC
  - Port 4: MacBook (usually off)
- Devices through switch: Wireless mouse + keyboard sharing a single USB dongle/receiver
- Issue is specific to this Dell — the switch works fine with all other machines

### Dell USB Ports
The Dell 7620 has two USB-A ports on different xHCI controllers:
- **Right port** → `PCI\VEN_8086&DEV_1135` — USB 2.0 controller; goes into Error state when switch reconnects; causes enumeration failures
- **Left port** → `PCI\VEN_8086&DEV_51ED` — USB 3.0 controller; stays OK; switch re-enumerates reliably

### Root Cause
Windows shows "Device Descriptor Request Failed" (VID_0000&PID_0002, problem code 43) when the switch reconnects to this laptop — the wireless receiver connects but fails USB enumeration. Confirmed BIOS bug in the Dell 7620.

### Fix Attempted (2026-06-17)
Installed the critical BIOS update via Dell Command Update:
- Update name: "Dell Inspiron 14 7420/16 7620 and Vostro 7620 System BIOS"
- Found under "other updates" (not the initial scan — had to click "check for other updates")

**Result: BIOS update did NOT fix the issue** — failed again on 2026-06-18.

### Failure Analysis (2026-06-18)
When failure occurs, two USB error entries appear on the Genesys Logic hub path:
- `VID_0000&PID_0002` — "Device Descriptor Request Failed" (same as before)
- `VID_0000&PID_0004` — "Set Address Failed" (new error type observed this session)
- The Intel xHCI controller `PCI\VEN_8086&DEV_1135` goes into **Error** state

The FixUSB scheduled task (`schtasks /run /tn "FixUSB"`) resets the xHCI controllers, which does bring the controller back to OK status. However, **the Genesys Logic hub (the USB switch itself) does not re-enumerate after the controller reset**. Directly connected devices (Samsung phone, webcam) come back fine; the switch does not.

**Implication**: The failure mode is not just the Logitech dongle failing enumeration — the entire USB switch hub stops responding after the xHCI controller is power-cycled. A physical unplug/replug of the switch cable is required to recover without a reboot.

### Workarounds / Fix
1. **Use the left USB port** — plug the switch cable into the left port (USB 3.0, DEV_51ED). This avoids the broken controller entirely and may be a permanent fix.
2. **Physical replug into left port**: If already on the right port and it fails, unplug from right and plug into left.
3. **FixUSB scheduled task**: Run `schtasks /run /tn "FixUSB"` in PowerShell (resets xHCI controllers) — only partially effective; still requires physical replug of the switch afterward.
   - Script at: `C:\Users\dalla\fix-usb.ps1`
4. **USB selective suspend**: Already disabled in power plan
5. **Emergency**: Plug wireless dongle directly into laptop USB port (bypasses switch)

### What Was Ruled Out
- USB switch hardware (works fine with other machines)
- USB cable / switch port (tried all combinations)
- Wireless dongle (works directly in laptop)
- Windows drivers (Windows built-in xHCI driver 10.0.26100.8521, fully up to date)
